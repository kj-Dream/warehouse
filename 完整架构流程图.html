SQL 执行

WHERE username='admin'
⬆ 返回
SysUser{id:1, username:"admin", password:"$2a$10$...", roleId:1, status:1}
返回数据流动：
数据库返回 SysUser → Mapper 封装成 Java 对象 → Service 继续处理其他业务 → 组装 LoginResponse（不含密码）→ Controller 用 Result 包装 → JSON 返回浏览器

    "id": 1,

    "realName": "系统管理员",
    "roleId": 1,
    "status": 1,
    "role": {"roleName": "管理员", "roleKey": "admin"}
    // ⚠️ 没有 password 字段！
  }
}

⏱️ 4. 三个典型请求的时序对比
💡 小图例： C = Controller  |  S = Service  |  M = Mapper  |  DB = 数据库
场景C → S → M → DB 各层做了什么
✅ 登录
POST /api/auth/login		 C：校验参数、验证码 → S：①查用户 ②比对密码 ③查状态 ④查角色 ⑤组装LoginResponse ⑥存Session → M：findByUsername findById → DB：SELECT sys_user + sys_role
→ 返回 {code:200, data:{id,username,realName,role}}
📥 创建入库单
POST /api/stock-in		 C：解析body+buildInDetails → S：①校验明细非空 ②计算总金额 ③生成单号 ④insert入库单 ⑤insert明细 → M：stockInMapper.insert + stockInDetailMapper.insert → DB：INSERT INTO stock_in + stock_in_detail
→ 返回 {code:200, msg:"入库单创建成功"}
✅ 审核通过
PUT /audit/stock-in/1/approve		 C：调Service → S：①updateStatus="approved" ②查明细 ③遍历明细调addStock → M：inventoryMapper.addStock → DB：INSERT ... ON DUPLICATE KEY UPDATE quantity=quantity+N
→ 返回 {code:200, msg:"入库单已审核通过，库存已更新"} 关键：审核通过才改库存！

📌 关键设计原则：
1. 三层架构：Controller(接参→校验→调Service) → Service(业务逻辑→事务) → Mapper(SQL操作)
2. RBAC 权限：用户 → 角色 → 菜单，不同角色登录看到不同菜单
3. 审核联动库存：审核通过后自动改库存（ON DUPLICATE KEY UPDATE），驳回不回滚
4. 数据脱敏：LoginResponse 不包含 password，Result 统一成功/失败格式
5. 统一返回：所有接口返回 {code, msg, data}，前端只判断 code 是否 200
